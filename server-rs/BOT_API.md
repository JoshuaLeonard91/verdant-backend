# Verdant Bot API

Verdant bots are server-scoped integrations. Bot delivery uses a dedicated
websocket gateway backed by Postgres `bot_event_outbox`; it does not consume
Redis pub/sub for bot event delivery.

## Token Scopes

Tokens are created from the existing authenticated server bot API:

`POST /api/servers/:serverId/bots/:botId/tokens`

Request body:

```json
{
  "name": "release-bot",
  "scopes": ["announcements:write", "feeds:read"],
  "allowedFeedIds": ["123"],
  "allowedChannelIds": []
}
```

Available scopes:

- `announcements:write`: create feed announcements through the bot REST API.
- `feeds:read`: receive feed announcement events from the bot gateway.
- `messages:write`: create structured card messages in allowlisted text channels.
- `messages:read`: receive allowed channel message lifecycle events.
- `message_content:read`: include message content in message events.
- `members:read`: receive member join/leave events.
- `audit:read`: receive audit log create events.

Feed and channel event delivery is allowlist-based. A token can only read
feed/channel events for IDs configured on the token, and token creation checks
that the creator can view those resources.

Feed announcement writes are also allowlist-based. A token with
`announcements:write` can only post to feeds listed in `allowedFeedIds`; an
empty feed allowlist has no implicit publish access.

Feed publish permission is still role-based after token validation. The bot
must have either a server permission role that can manage the server, or one of
the bot's assigned access roles must be listed in the target feed's
`publishRoleIds`. In practice, create a dedicated role such as `Bot Publisher`,
assign it to the bot, and add that role to each feed the bot is allowed to
publish into. `allowedFeedIds` scopes the token; `publishRoleIds` scopes the
server-side feed permission.

## REST API

`GET /api/bot/me`

Requires `Authorization: Bot <token>` and returns the bot identity, scopes,
allowlists, and gateway URL.

`POST /api/bot/feeds/:feedId/announcements`

Requires `Authorization: Bot <token>` and `announcements:write`. The body uses
the same announcement JSON as the user-facing feed announcement endpoint.
The token must include the target feed in `allowedFeedIds`. Provide
`Idempotency-Key` for release automation retries.

Structured feed cards are Flutter-native JSON, not HTML. The backend
deserializes with unknown-field rejection, sanitizes all text fields, validates
URLs and server-local button targets, and enforces bot scope/role/feed
allowlists before inserting the announcement.

Example request body:

```json
{
  "title": "Programmatic feed card",
  "titleStyle": { "fontSize": 20, "weight": "semibold" },
  "description": "Posted through the bot REST API.",
  "descriptionStyle": { "fontSize": 14 },
  "color": "#1ee3b6",
  "sections": [
    {
      "type": "richText",
      "spans": [
        { "text": "Status: " },
        {
          "text": "healthy",
          "style": { "color": "#22c55e", "weight": "bold", "italic": true }
        }
      ]
    },
    {
      "type": "chart",
      "title": "Audience split",
      "kind": "donut",
      "points": [
        { "label": "Desktop", "value": 58 },
        { "label": "Mobile", "value": 24 },
        { "label": "Web", "value": 18 }
      ]
    },
    {
      "type": "video",
      "url": "https://www.youtube.com/watch?v=k1_ODDevbY8",
      "videoId": "k1_ODDevbY8",
      "title": "Walkthrough"
    },
    {
      "type": "button",
      "label": "Join server",
      "style": "secondary",
      "action": { "type": "invite", "code": "Verdant123" }
    }
  ],
  "footer": "Generated by a bot",
  "footerStyle": { "fontSize": 12 }
}
```

Supported section types are `text`, `richText`, `heading`, `quote`, `list`,
`table`, `code`, `image`, `divider`, `button`, `chart`, and `video`. The
server also accepts legacy section type `youtube` as an alias for `video`.
Chart kinds are `bar`, `line`, `donut`, `metrics`, `progress`, and
`sparkline`.

`POST /api/bot/channels/:channelId/cards`

Requires `Authorization: Bot <token>` and `messages:write`. The body uses the
same structured card JSON as feed announcements, and the token must include the
target channel in `allowedChannelIds`. This posts a rich card into a normal text
channel for bot-authored summaries such as rankings, match results, release
notices, or moderation reports. Provide `Idempotency-Key` for retryable jobs.

## Gateway

Connect to `/bot-gateway`, then identify:

```json
{
  "op": "IDENTIFY",
  "d": {
    "token": "verdant_bot_token",
    "intents": ["FEEDS", "MESSAGES", "MEMBERS", "AUDIT_LOG"],
    "serverIds": ["123"],
    "lastEventId": "456"
  }
}
```

If `lastEventId` is omitted, the gateway starts after the latest stored event
for the bot server. Events are delivered as:

```json
{
  "op": "DISPATCH",
  "t": "MESSAGE_CREATE",
  "s": "457",
  "d": {}
}
```

Use `RESUME` with `lastEventId` to replay from a known event ID. The outbox
retains events for 7 days; idempotency keys retain for 30 days.

## Release Bot

Client release uploads can post a feed announcement when both env vars are set:

- `RELEASE_BOT_TOKEN`
- `RELEASE_BOT_FEED_ID`

Optional:

- `RELEASE_BOT_SERVER_URL`, defaults to `NOTIFY_SERVER_URL` or
  `https://api.verdant.chat`.

The server Docker workflow uses the same token/feed variables from GitHub
secrets/vars to post image publish announcements.

## Local Smoke Test

From the repo root, a scoped bot token can publish the current Flutter feed card
schema with:

```bash
VERDANT_BOT_TOKEN=... \
VERDANT_BOT_FEED_ID=... \
VERDANT_API_URL=https://api.verdant.chat \
bun run --filter @verdant/bot smoke:feed
```

Optional env vars are `VERDANT_INVITE_CODE`, `VERDANT_YOUTUBE_URL`, and
`VERDANT_BOT_IDEMPOTENCY_KEY`. Do not print bot tokens in shell logs or CI
output.

Before running the smoke test against a real server, verify:

- The token has `announcements:write` and the feed ID is present in
  `allowedFeedIds`.
- The bot has a server access role that appears in the target feed's
  `publishRoleIds`.
- Any `navigateChannel` button target is a text channel in the same server.
